CN352 1.0003 MUST BT1034 Configure Additional Adapter applctn instances to provide cross-site resilienceCurrently, the adapter components in the messaging infrastructure are configured to provide an amount of resilience to failure, such that it is possible to lose one of the adapter servers without leaving any of the application request queues unserviced (i.e. no consumer processing messages from the queue). This is restricted within a site, i.e. LiveA and LiveB, such that if all of the adapter facility was lost in one site, then the application request queues would cease to be serviced. The purpose of this CCN is to propose the configuration of additional, dormant adapter instances on both LiveA and LiveB, solely to use in the event of the loss of the whole adapter facility within a site. A new instance for each of the applications (PDS, ACF, LRS) will be configured. By default they will not be active. They will be configured such that the queue destinations that they will consume from are those made available by the broker on the opposite site, e.g. in LiveA, they will be configured to point at the LiveB queue manager cluster and vice versa. In the event of a problem, these instances could be started (selectively) with the sole purpose of clearing down the queues on the site experiencing the problem. They would be expected to be active long enough for the queues to be fully emptied, on the assumption that part of the problem attack would be to suspend the top-end services such that the affected messaging channels are routed to the fully operational site. It has been suggested that there are no design issues with this proposed solution. It remains to be understood whether or not there are any infrastructural or network design issues with the proposed solution.

CN499 1.0009 MUST BT1449 NPFIT-NCR-CCN-0404 Implementation of Standard CDA Document Structure in 2008-AThis CCN proposes that CFH should implement a standard Clinical Document Architecture (CDA) document structure for the 2008-A spine release. Making this change will realise substantial clinical and financial benefits and mitigate a number of significant clinical and reputational risks. NB. The opportunity to make this change has only been available since the approach to sealing changed from statement level to document level sealing (CCN387) and the change outlined here represents the approach that would have been adopted from the outset, but for the requirement to have â€œin-documentâ€� statement-level sealing.

Description of changes

The proposed change has three aspects (supporting information is provided in the form of a Slide Deck (CDAProposal.ppt (including indicative RMiMs )):

a. Simplification of Text Sections
Currently, the 2008-A CDA documents are broken into atomic information components, each of which may be coded or non-coded, but all of which must have a title section. This is to enable patients to seal part(s) of a document. The consequence of this approach is that the document becomes fragmented and the different fragments cannot be rendered in a way that preserves a natural document format.

The proposal is to replace this atomic structure with a unified structure that makes it a simple matter to construct clinical documents in a clinically natural and familiar form. The unity of the structure will be provided by the use of one or more text sections that may or may not have multiple coded templates and CRE types associated with them. The key point is that the use of coded templates and indexing against CRE types would function at least as effectively as on the current approach, but would not impose an artificial and unnecessarily fragmented structure upon clinical documents.

The contractual documentation impacted by this change is MiM 6.2 Draft B and the change would affect the RMiMs, Example Messages, Schemas, Schematrons, Tabular Views for all CDA domains (including PSIS Queries and HealthSpace). It is anticipated that the full set of MiM changes would take 2-3 weeks.

By making this change we will greatly increase the flexibility of the CDA Documents and make it possible to construct clinical documents that have a natural narrative format. The documents will better support clinical practice and will make it significantly easier for end systems to render CDA documents.


b. Removal of Business Sections
Currently, the CDA documents contain 4 business sections. These perform no semantic function within the message and are used merely at the presentation level to provide some high-level groupings to the fragmented documents. Removing them is a trivial change that has been agreed by clinical advisers. The benefit of making this change is that it allows greater flexibility in the use of the CDA documents since no presentation level grouping would be imposed on the documents. In this way it would be a simple matter for the CDA documents to support directives and guidance provided by the Royal Colleges and other professional bodies (e.g. the RCP work on Standardised Clinical Documentation).


c. Inclusion of Encompassing Encounter within CDA Clinical Documents
Currently, the CDA documents do not make use of Encompassing Encounter (a standard component of CDA), instead they use the Care Event template to describe the type of care provided and the setting within which care was delivered. This is less than ideal, since it does not allow clinicians to distinguish between the overall encounter type and specific care events that take place within that encounter. It is proposed that we use Encompassing Encounter within the clinical CDA documents and this is the approach that would have been taken had the clinical use of the component been properly understood at the outset. This is a trivial change that is well supported by clinical advisors and the supplier community.

Background

The current baselined requirements defining the SDS Organisation Administration Service (OAS) functionality state that the System must allow the User to submit a request to re-parent one or more organisations by specifying the current parent organisation and the new parent organisation. The selection of the new parent organisations is restricted to organisations of the same type as the current parent. The reason for this restriction was to avoid complications that could arise as a result of the inherited characteristics, should the new parent template type definition vary from that of the current parent template type definition.

The main template type definitions that could give rise to complications are:

" The mandatory associations inherited from the parent organisation.
" The RA Category of the Parent Organisation.
" The permitted parent types for the children of a particular organisation may vary making it impossible to re-parent all the child organisations to the same parent organisation.

NHS CFH has stated that the baselined requirements are too restrictive and would prevent them from carrying out mandated organisation reconfigurations. For instance, in the event that a Primary Care Trust (PCT) type organisation converts to a Care Trust type organisation all the PCT's children would effectively need to be re-parented to the new Care Trust organisation set up to replace the PCT. As it stands the application functionality would not allow this operation.

Requirement

CfH have requested that the OAS functionality is modified as follows:

1. Allow particular organisation types to be grouped by the Authority for the purposes of the bulk re-parent request. There is no requirement for these groupings to be configurable and there will be no facility covered by this CCN to add or remove groupings from those specified against this CCN.

2. OAS must allow organisations belonging to a parent organisation of a type included in one of these groups to be re-parented to any organisation of a type specified in the same group.

3. OAS must validate at submission of the bulk re-parent request that if the organisation types of the current and new parent differ, the two organisation type are still compatible in terms of the rules of association and permitted child types. In the case where the organisation types are not compatible, the System must display an appropriate error message and prevent the re-parent request from being submitted.

In the example of the PCT/Care Trust scenario, if the current parent is of type PCT, the System would permit the User to choose the new parent organisation from organisations of type PCT or Care Trust. If the current parent type was Care Trust, the System would allow the User to select the new parent from PCT or Care Trust type organisations.

Clarifications

CN543 1.0013 MUST BT1480 Re-Postcoding of AddressesBackground

There are times when the Royal Mail has to make large-scale changes to postal addresses. This may be when new building developments such as housing estates or business parks within already densely populated inner city areas leads to 'Postcode exhaustion' where we have no free Postcodes to allocate. However, due to the geographical nature of the Postcode system the wider area is considered and new Postcode sectors and Postcodes are allocated. This may also apply in the cases of demolished buildings.

There is a changeover period of 12 months allowed by the Royal Mail, where both old and postcodes are valid.

The Royal Mail produces a 6-monthly summary of changes known as a Postcode Update which are published to the internet.

http://www.royalmail.com/portal/rm/jump2?catId=400084 and mediaId=600127

The Postcode Update File (PUF) contains postcode, locality and thoroughfare update. Postcode updates may involve either changing or splitting a postcode.

Requirements

This CCN introduces a requirement to provide a periodic update to PDS based on the PUF. This has been elaborated as follows:

1. A facility is required to allow Service to update PDS based on the contents of a PUF.

2. Addresses in PDS with a postcode matching a changed postcode should have their postcode changed to the new postcode.

3. Addresses in PDS with a postcode matching a split postcode should have their postcode changed to the new postcode if the delivery point details provided in the PUF can be unambiguously matched with the PDS address.

4. PUF postcodes changes should be performed using standard PDS update processing and should generate Demographic Notifications. (Sent to NHS Blood Donor Register, NHS Organ Donor Register, 2008-B accredited Primary Care Provider systems and Stage 4 NHAIS systems.)

5. PUF postcode changes should not cause a Review Dispensing Authorisation Reason work item to be raised with DSA.

6. Service should obtain the PUF and run the postcode update facility on a six monthly basis.

1. It is assumed that there is no requirement to process the Locality or Thoroughfare update contained in the PUF e.g. Abbey Town changing to Abbeytown.

2. It is assumed that the PUF file format will not be changed by the Royal Mail. Should the format change it is assumed that CfH will raise another CCN to cover any changes required to the postcode update facility.

This approach significantly affects BTs? ability to accurately map the originator of the message to meaningful NHS/NPfIT geographical areas for SLA reporting.

3. In addition, we are receiving an increasing number of requests to report on Spine performance by business function (i.e. response times for ETP, C and B, GP-GP, PDS), and we do not currently receive routine reports in this format from BT to support these requests.

Consequence of no change:
1. An element of Spine SLA reporting becomes naturally obsolete and of little relevance to the business supported by this Programme.
